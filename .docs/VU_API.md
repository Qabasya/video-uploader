# VU_API — API fs-video-uploader и интеграция с сайтом (fs-lms)

Название файла — по аналогии с `FS_LMS_API.md` в репозитории плагина (`VU` = Video Uploader). Здесь — обе стороны контракта:

1. **Входящий HTTP API** — то, что этот сервис предоставляет сам (для мониторинга/техперсонала).
2. **Исходящий REST** — то, чем сервис стучится в fs-lms при регистрации видео.
3. **Побочный канал через S3** — метаданные, которые плагин может прочитать напрямую из бакета, не спрашивая сервис.

Реализация: `video_uploader/api/app.py` (входящий API), `video_uploader/lms/client.py` (исходящий REST), `video_uploader/storage/s3_gateway.py` (манифест + `x-amz-meta-*`).

- Базовый URL входящего API: `http://<host>:<API_PORT>` (по умолчанию порт `8090`, см. `.env`).
- Аутентификации на входящем API нет — доступ ограничен сетевым сегментом (Tech/Admin), не приложением.
- Формат — везде `application/json`.
- Синхронные обработчики: FastAPI сам уводит их в threadpool (внутри — блокирующие вызовы к SQLite и `threading.Event`).

## 1. Входящий HTTP API (этот сервис)

### `GET /health`

Проверка живости сервиса — используется Docker `HEALTHCHECK` в `docker-compose.yml`.

**Запрос**

```bash
curl -s http://localhost:8090/health
```

**Ответ `200 OK`**

Сразу после старта, до первого цикла сканирования:

```json
{
  "status": "ok",
  "last_scan_at": null
}
```

После хотя бы одного цикла — `last_scan_at` в ISO 8601, UTC:

```json
{
  "status": "ok",
  "last_scan_at": "2026-07-17T13:44:12.405483+00:00"
}
```

`status` всегда `"ok"`, если процесс жив и ответил — эндпоинт не проверяет доступность S3/LMS/шары отдельно, это было бы дороже, чем нужно для liveness-проверки контейнера. Диагностика конкретных проблем — `GET /status` и логи (Loki/`DATA_DIR/logs/uploader.log`).

### `GET /status`

Счётчики по статусам реестра + последние записи — для быстрой диагностики без похода в SQLite напрямую.

**Запрос**

```bash
curl -s http://localhost:8090/status
```

**Ответ `200 OK`**

Поля:

| Поле | Тип | Описание |
|---|---|---|
| `counts` | `object` | Ключ — один из статусов реестра (`discovered`, `uploading`, `uploaded`, `registered`, `archived`, `failed`, `skipped_old`, `skipped_unmapped`), значение — количество записей. Статусы, по которым записей нет, в объекте отсутствуют (не `0`). |

| `recent[].mtime` | `number` | Unix-время (эпоха), как есть из `os.stat()`. |
| `recent[].created_at`, `.updated_at` | `string` | UTC, ISO 8601. |

Пустой реестр (сервис только что запущен, файлов ещё не было) — `{"counts": {}, "recent": []}`.

### `POST /rescan`

Триггер внеочередного цикла сканирования, не дожидаясь `SCAN_INTERVAL_SECONDS`.

**Запрос**

```bash
curl -s -X POST http://localhost:8090/rescan
```

Тело запроса не требуется и игнорируется.

**Ответ `200 OK`**

```json
{
  "status": "triggered"
}
```

Важно: ответ приходит немедленно и **не означает**, что скан уже завершился — это именно триггер («разбудить» фоновый поток пораньше), а не синхронный запуск. Единственный поток, который когда-либо выполняет цикл сканирования, — фоновый воркер; `/rescan` не запускает его параллельно, а прерывает текущее ожидание между циклами. Результат смотреть через `GET /status` спустя несколько секунд или в логах.

Повторные вызовы `/rescan` во время уже идущего цикла безопасны — не создают очередь и не запускают несколько сканирований одновременно.

## 2. Исходящий REST: регистрация видео в fs-lms

Сервис сам выступает клиентом REST API плагина fs-lms (`video_uploader/lms/client.py`) — это вызов наружу на шаге `register` пайплайна, а не эндпоинт этого сервиса.

### Аутентификация — HMAC

Единая схема интеграций fs-lms (как у модуля AdSync); статического токена в заголовках нет.

```
POST {LMS_BASE_URL}/wp-json/fs-lms/v1/videos
Headers:
  X-Fs-Timestamp: <unix-время, секунды>
  X-Fs-Signature: hex( hmac_sha256( "{timestamp}.{raw_body}", LMS_HMAC_SECRET ) )
  Content-Type: application/json
```

- `raw_body` — байты JSON-тела ровно в том виде, в каком отправлены (подпись считается от них, JSON сериализуется один раз и переиспользуется для подписи и отправки).
- Плагин отвергает запрос (`401`), если `|now − timestamp| > 300 c` (анти-replay) или подпись не сошлась → **часы LXC должны быть синхронизированы (NTP)**.
- Секрет: env `LMS_HMAC_SECRET` этого сервиса = константа `FS_LMS_VIDEO_HMAC_SECRET` в `wp-config.php` плагина. Значение общее для обеих сторон, отдельное от секрета AD-синка.

### Тело запроса

```json
{
  "s3_bucket": "f6bcd57c2800-fs-video",
  "s3_key": "videos/kege-1/2026/07/2026-07-08_16-04_6a1bf35f.webm",
  "manifest_key": "videos/kege-1/2026/07/2026-07-08_16-04_6a1bf35f.webm.json",
  "group_slug": "kege-1",
  "lms": {"group_id": 3, "course_id": 42, "teacher_id": 7},
  "recorded_at": "2026-07-08T16:04:45+03:00",
  "size_bytes": 245681302,
  "sha256": "6a1bf35f2da20adaa77617f3512f275f1d1e3beaf9ac79bdb88fd8b203683041",
  "duration_sec": null
}
```

- `lms`-блок пробрасывается из `config/groups.yaml` как есть, без интерпретации сервисом:
  - папка учебной группы → `{group_id, course_id, teacher_id}`;
  - персональная папка преподавателя (индивидуальные занятия) → только `{teacher_id}`, без `group_id`.
- **Семантика состава — на стороне плагина**: наличие `group_id` включает групповую ветку резолва (видео крепится к занятию группы по дате/времени `recorded_at`); только `teacher_id` без `group_id` — индивидуальную ветку (занятие `kind='individual'` этого преподавателя). `course_id`/`teacher_id` в групповой ветке — не параметры резолва, а кросс-чек `groups.yaml` против таблицы `fs_lms_groups` (обнаружение рассинхрона конфигов); сам матчинг группы идёт только по `group_id`.
- `teacher_id` — числовой WP `user_id` (тот же, что в `fs_lms_groups.teacher_id`), не логин.

### Ответ плагина

`200 OK`:

```json
{"ok": true, "matched": true, "group_lesson_id": 128}
```

или

```json
{"ok": true, "matched": false, "group_lesson_id": null}
```

- `matched: false` — **не ошибка**: видео зарегистрировано (upsert по `s3_key` прошёл), но подходящее занятие по дате/времени не найдено — плагин оставил видео на ручную привязку администратором. Для сервиса оба случая = статус `registered`; `matched: false` логируется как WARNING (сигнал о рассинхроне расписания, не о поломке).
- Идемпотентность — на стороне плагина: upsert по `s3_key`, повторная отправка одного и того же видео безопасна.

### Обработка ошибок и ретраи

| Ответ | Итог | Поведение |
|---|---|---|
| `200`/`201` | `registered` | Успех, см. `matched` выше |
| `5xx`, сетевая ошибка/таймаут | остаётся в очереди | Ретрай на следующих циклах сканирования, с экспоненциальным backoff, до `MAX_ATTEMPTS` |
| Прочие `4xx` (включая `401` — просроченная метка времени/неверная подпись) | `failed` | ERROR в лог, без ретраев — нужно вмешательство (обычно рассинхрон часов или неверный `LMS_HMAC_SECRET`) |

### DRY_RUN

Пока REST-эндпоинт плагина не готов (или для локальной проверки) — `DRY_RUN=true`: шаг `register` логируется и засчитывается успешным, реального запроса к `LMS_BASE_URL` не будет. То же самое применяется к загрузке в S3 и архивации — весь путь с внешними побочными эффектами подменяется заглушками в composition root (`main.py`).

## 3. Побочный канал: метаданные в S3 (без похода в сервис)

Плагин может свериться с данными видео напрямую в бакете — не запрашивая ничего у `fs-video-uploader`.

### Манифест `{video_key}.json`

Кладётся рядом с каждым видео, `ContentType: application/json`, UTF-8:

```json
{
  "schema": 2,
  "group_slug": "kege-1",
  "source_folder": "КЕГЭ-1",
  "lms": {"group_id": 3, "course_id": 42, "teacher_id": 7},
  "recorded_at": "2026-07-08T16:04:45+03:00",
  "original_name": "Встреча_в_Телемосте_08_07_26_16_04_45_—_запись.webm",
  "size_bytes": 123456789,
  "sha256": "…",
  "uploaded_at": "2026-07-14T21:00:05+00:00",
  "service": {"name": "fs-video-uploader", "version": "0.1.0"}
}
```

### `x-amz-meta-*` объекта видео

Только ASCII (кириллица, включая `source_folder`, — только в манифесте, не в метаданных объекта):

- `x-amz-meta-group-slug`
- `x-amz-meta-recorded-at`
- `x-amz-meta-sha256`
- `x-amz-meta-lms-<key>` — зеркало каждой пары из блока `lms` (символ `_` в имени ключа заменяется на `-`, например `lms.teacher_id` → `x-amz-meta-lms-teacher-id`)

Так плагин получает все ID одним `HEAD`-запросом к объекту, не скачивая манифест целиком.

### Выдача видео ученикам

Раздача студентам (presigned URL) — целиком на стороне плагина, `fs-video-uploader` в этом не участвует и не знает о факте просмотра. Бакет приватный; presigned-ссылки генерирует модуль `VideoLibrary` плагина по требованию.

## 4. Что определяет состав `lms`-блока

Источник — `config/groups.yaml` на этой стороне. Формат, валидация и связь с БД плагина (`fs_lms_groups`) — `.docs/basic_doc.md`, раздел 5, и `.docs/CLAUDE.md`, раздел `config/groups.yaml`. Сервис трактует блок `lms` как непрозрачный (opaque) плоский словарь — валидирует только формат ключей/значений, не их смысл; смысл полностью на стороне fs-lms.
